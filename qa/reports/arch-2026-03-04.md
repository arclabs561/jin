### 3b. ANNIndex Trait Design

Defined in `src/ann/traits.rs`:

```rust
pub trait ANNIndex {
    fn add(&mut self, doc_id: u32, vector: Vec<f32>) -> Result<(), RetrieveError>;
    fn add_slice(&mut self, doc_id: u32, vector: &[f32]) -> Result<(), RetrieveError>;
    fn build(&mut self) -> Result<(), RetrieveError>;
    fn search(&self, query: &[f32], k: usize) -> Result<Vec<(u32, f32)>, RetrieveError>;
    fn size_bytes(&self) -> usize;
    fn stats(&self) -> ANNStats;
    fn dimension(&self) -> usize;
    fn num_vectors(&self) -> usize;
}
```

Implementations: HNSW, NSW, SNG, DiskANN, SCANN, IVF-PQ, KD-Tree, Ball-Tree, K-Means-Tree, RP-Tree (10 total). All feature-gated.

**Findings**:

- **F-03 [COHERENCE]**: `ANNIndex::search()` takes only `(query, k)` but HNSW and NSW have `ef_search` as a parameter. The trait impl hardcodes `self.params.ef_search`. Users who want per-query `ef_search` override must downcast to the concrete type, losing the abstraction.
- **F-04 [SURFACE]**: `doc_id` is locked to `u32`. This limits to ~4B unique IDs. For billion-scale use cases this is tight but acceptable for now.
- **F-05 [SURFACE]**: `add()` takes `Vec<f32>` (owned), while `add_slice()` takes `&[f32]`. The default `add_slice` impl allocates then calls `add`. Indexes that override both (HNSW, NSW, DiskANN) are fine. Indexes that don't override `add_slice` (SNG, KD-Tree, Ball-Tree, RP-Tree, K-Means-Tree) get an unnecessary allocation on the `add_slice` path.

## 4. Distance Metric Coherence

### 4a. Per-Index Distance Functions

| Index | Build Distance | Search Distance | Metric |
|-------|---------------|-----------------|--------|
| HNSW | `hnsw::distance::cosine_distance` (= `1 - dot(a,b)`) | `hnsw::distance::cosine_distance` | Cosine (assumes L2-normalized) |
| NSW | `1.0 - simd::dot(a, b)` | `1.0 - simd::dot(a, b)` | Cosine (assumes L2-normalized) |
| SNG | `1.0 - simd::dot(a, b)` | `1.0 - simd::dot(a, b)` | Cosine (assumes L2-normalized) |
| DiskANN | `sum((a[i]-b[i])^2)` | `sum((a[i]-b[i])^2)` | **L2 squared** |
| Vamana | `hnsw_distance::cosine_distance` | `hnsw_distance::cosine_distance` | Cosine (assumes L2-normalized) |
| IVF-PQ | PQ ADC tables (L2 on residuals) | PQ ADC tables | L2 on residuals |
| ScaNN | Inner product on centroids | Inner product + reranking | **Inner product (MIPS)** |
| Streaming buffer | `crate::distance::l2_distance` | `crate::distance::l2_distance` | **L2** |

**Findings**:

- **F-07 [STRUCTURAL]**: **DiskANN uses L2 squared while all other graph indexes use cosine (1 - dot)**. Switching from HNSW to DiskANN with the same vectors produces incomparable results. The `ANNIndex` trait returns `(u32, f32)` where `f32` is the distance, but the semantics of "distance" differ across implementations. This is not documented on the trait.
- **F-08 [STRUCTURAL]**: **Streaming buffer uses `crate::distance::l2_distance`** while the graph indexes it wraps (HNSW, NSW) use cosine distance. When `StreamingCoordinator` merges buffer search results with main index results, the distances have different semantics. Results from buffer (L2) and main index (cosine) are incomparable.
- **F-09 [COHERENCE]**: The `DistanceMetric` enum exists in `src/distance.rs` but none of the indexes actually use it. Each index hard-codes its distance function. The enum is only used by external evaluation utilities.
- **F-10 [COHERENCE]**: DiskANN implements its own inline `dist()` function (`src/diskann/graph.rs:300-302`) instead of using `crate::distance::l2_distance` or `crate::simd`. This bypasses SIMD acceleration.

### 4b. Normalization Contract

- HNSW, NSW, SNG, Vamana all compute `1 - dot(a,b)` which is only a valid cosine distance **for L2-normalized vectors**.
- **No runtime check** on normalization at `add()` or `add_slice()` time in any index.
- The `distance.rs` module documents this: "HNSW uses dot-product-based cosine distance for speed and therefore expects inputs to be L2-normalized."
- Individual index doc comments do **not** state the normalization requirement.

**F-11 [SURFACE]**: No index documents its normalization requirement on `add()` or `add_slice()`. Users must read the distance module docs to discover this.

## 6. Error Handling

### 6a. Error Hierarchy

Three error types:

| Type | Location | Role |
|------|----------|------|
| `RetrieveError` | `src/error.rs` | Core indexing/search errors (10 variants) |
| `PersistenceError` | `src/persistence/error.rs` | Disk persistence errors (10 variants) |
| `CompressionError` | `src/compression/error.rs` | Compression operations (5 variants) |

**Conversions**:
- `std::io::Error -> RetrieveError` (via `From`)
- `postcard::Error -> PersistenceError` (feature-gated)
- `bincode::Error -> PersistenceError` (feature-gated)
- `hiqlite::Error -> PersistenceError` (feature-gated)
- `durability::PersistenceError -> PersistenceError` (feature-gated)
- **No `From<PersistenceError> for RetrieveError`** -- these are separate error hierarchies.
- **No `From<CompressionError> for RetrieveError`** -- compression errors get mapped via `map_err` with `RetrieveError::Other`.

### 6b. Unwrap/Expect Census

**Total**: 195 occurrences across 35 files.

Breakdown of **non-test** occurrences (estimated from context analysis):

| File | Non-test unwrap/expect | Severity |
|------|------------------------|----------|
| `ivf_pq/search.rs:205` | 1 (`decompressed_cache.as_ref().unwrap()`) | Library code, panics if cache logic has bug |
| `classic/trees/balltree.rs:171,174` | 2 (`right_indices.pop().unwrap()`) | Library code, guarded by `is_empty()` check above |
| `persistence/segment.rs:103` | 1 (`postings.get(*term).unwrap()`) | Library code, panics if term not in postings |

The vast majority of unwrap/expect calls (~190) are in `#[cfg(test)]` blocks or test modules, which is acceptable.

**F-14 [SURFACE]**: `ivf_pq/search.rs:205` uses `.unwrap()` on a cache reference in library code. If the decompression cache logic has a bug, this panics in production search.

### 6c. Error Variant Misuse

**F-15 [COHERENCE]**: `RetrieveError::EmptyQuery` is used for `dimension == 0` validation in constructors:
- `src/scann/search.rs:61` -- `SCANNIndex::new()` returns `EmptyQuery` for zero dimension
- `src/ivf_pq/search.rs:254` -- `IVFPQIndex::new()` returns `EmptyQuery` for zero dimension
- `src/classic/trees/rp_forest.rs:99` -- `RpForestIndex::new()` returns `EmptyQuery` for zero dimension

These should use `RetrieveError::InvalidParameter`. Other indexes (HNSW, NSW, SNG, DiskANN) correctly use `InvalidParameter` for this.

**F-16 [COHERENCE]**: `RetrieveError::Other(String)` is heavily used as a catch-all. 120+ usages across the codebase. Many represent distinct error conditions (e.g., "Index not built", "Cannot add after build", "m must be > 0") that would benefit from dedicated variants. This makes programmatic error handling difficult for downstream users.
